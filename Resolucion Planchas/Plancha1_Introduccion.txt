PREGUNTA 1: POR QUÉ SE PREFIERE EMULAR UNA CPU EN VEZ DE UTILIZAR DIRECTAMENTE LA CPU EXISTENTE?

    Emular una CPU garantiza que trabajemos sobre una arquitectura suficientemente
    sencilla para la cursada y nos deja ocultar detalles de bajo nivel que no aportan al objetivo pedagógico. 

    Al tener el control del hardware simulado, podemos instruir al sistema para forzar interrupciones, excepciones o fallos
    sin arriesgar el equipo físico y con resultados reproducibles en cualquier PC.

    Esa capa de abstracción protege al anfitrión y ayuda a concentrarnos en la lógica
    del sistema operativo: planificación, gestión de memoria o manejo de excepciones, sin distraernos con la complejidad de x86-64.

    En definitiva, la emulación brinda un laboratorio seguro y didáctico donde cada concepto se puede ilustrar con precisión y sin depender de configuraciones
    específicas del procesador real.

PREGUNTA 2: CUÁNTA MEMORIA TIENE LA MÁQUINA SIMULADA PARA NACHOS?

    La cantidad de memoria física simulada viene dada por la constante MEMORY_SIZE = NUM_PHYS_PAGES * PAGE_SIZE. 

    El Nachos fija NUM_PHYS_PAGES = 32 y PAGE_SIZE = SECTOR_SIZE, donde el SECTOR_SIZE es de 128 bytes.

    Entonces la máquina emulada tiene 32 páginas de 128 bytes, es decir, 4.096 bytes de memoria disponibles para los procesos de usuario.

PREGUNTA 3: ¿QUÉ MODIFICARÍA PARA CAMBIAR LA CANTIDAD DE MEMORIA?

    -Para modificar la cantidad de memoria, entendemos que lo mas correcto sería simplemente variar el numero de paginas (NUM_PHYS_PAGES).

    -Otra forma sin tocar el código es correr nachos -m N, que hace que el kernel cree N páginas físicas. 
        No es ideal porque depende de cada corrida y no queda documentada en la configuración fija.

PREGUNTA 4: DE QUÉ TAMANIO ES UN DISCO?
    El tamanio del disco viene determinado por la cantidad de sectores que tiene. Esto sería: NUM_SECTORS = SECTORES_PER_TRACK * NUM_TRACKS.
        Luego SECTORES_PER_TRACK = 32 y NUM_TRACKS = 32, por lo que NUM_SECTORS = 1024
    
    Por lo que el disco tiene tamaño: NUM_SECTORS * SECTOR_SIZE = 1024 * 128 = 131072 bytes

PREGUNTA 8: QUÉ EFECTO HACEN LAS MACROS ASSERT Y DEBUG DEFINIDAS EN lib/utility.hh?
    ASSERT: es la macro definida para validar suposiciones.
        Si la condición falla imprime el predicado, el archivo, la línea y aborta el programa.
    
    DEBUG se encarga de imprimir mensajes de depuracion. Verifica si una bandera esta
        habilitada y en tal caso muestra la informacion correspondiente.

PREGUNTA 9: COMENTE EL EFECTO DE LAS DISTINTAS BANDERAS DE DEPURACION:
    * `+` -- turn on all debug messages.
    * `t` -- thread system.
    * `s` -- semaphores, locks, and conditions.
    * `i` -- interrupt emulation.
    * `m` -- machine emulation (requires *USER_PROGRAM*).
    * `d` -- disk emulation (requires *FILESYS*).
    * `f` -- file system (requires *FILESYS*).
    * `a` -- address spaces (requires *USER_PROGRAM*).
    * `e` -- exception handling (requires *USER_PROGRAM*).
